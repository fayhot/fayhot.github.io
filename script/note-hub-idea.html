Note Hub Idea(关于笔记知识库的一些想法)

Posted on 2020-07-07 In 笔记 Views:

基于笔记的知识库 [notehub.com vs github.com]

前言

我尽量用简单的方式来描述这个产品想法的心路历程和客观环境。这个产品用户需求的本质是

  1. 新技术的应用会推动社会的进步并带来便利，进而取代原有的解决方案成为新的事实标准。
  2. 以人为中心的设计原则，所有人公开的笔记都可以归入自己的内容列表。 从而构建自己是知识库。

    当前主流的社区社交网络是以内容为核心来设计，而IM相关的社交网络是以人为中心的原则来设计的。
    因为只有持续强工具属性的社交网络才有资格以人为中心做设计。

定位

  1. notehub.com 是一个以人为中心而构建的一个知识库。

包含了用户感兴趣的内容以及过去学习工作生活中不可缺少的相关内容。

  1. notehub.com 包括生产力工具、NoteFeed Flow两个场景。

以新技术标准的组合应用来完成feed的统一（即一致化）生产和消费，包括文字、图片、视频。

要素与环境

解构、定义 notehub.com 是什么、做什么、怎么做，以及当前的客观环境。

  1. 基于markdown的笔记编辑工具。

    当前市场环境

    1. blog服务逐次关闭，只剩下简书、美篇等少数平台仍然提供“写点东西”的服务。
    2. 印象笔记、Bear 等等软件都提供了非常优秀的富媒体编辑、排版、同步的一体化方案。
    3. 手机和PC上自带的备忘录，虽然很粗糙，但是能用，很多临时性的场景凑合凑合是可以的。

  2. NoteFeed Flow的流动机制。

    1. 纯粹的笔记生成工具没有产业价值，甚至缺乏普遍的强用户认同感，导致付费敏感。
    2. 基于类似 git fork 机制，复制笔记到用户的个人笔记列表，实现最基本的数据分享。

    应该是需要一个新的 名词 来描述 fork 这种共享机制的，但是我没有想到合适的词。
    现在常见的收藏和书签机制都是一种"软链接"的指向模式。应该被更合适的方案取代。

    1. 需要跳转到笔记源头来查重和回溯内容。
    2. 产品逻辑和用户体验上都造成巨大的断层，
    3. 非常不利于笔记数据知识库的搜索场景。
    4. 违背以人为中心的设计原则，用户对收藏的笔记，缺乏心理认同。
    1. 基于推荐体系的公开笔记Feed流动。

    有足够多的笔记内容时，需要通过推荐机制来解决Feed分发的问题。学习相关的图文笔记，美妆相关的图文笔记，旅游相关的图文笔记，摄影相关的图文笔记 等等诸多场景下的不同类别和领域的笔记，需要推荐机制来确保流动到不同的人群。

    1. 基于搜索体系的长尾知识挖掘。

    账号体系、社交网络、移动化未普及的年代，搜索引擎通过爬虫存储了近乎全网络的数据内容或摘要。基于此，实现了以搜索行为为中心节点的网络互连。推荐之外，用户仍然需要搜索来解决对应的庞大体量的长尾需求。

  3. 冷启动与扩张

    1. Markdown 编辑器
    1. 好用的Markdown编辑器（Web || Mac || Windows），就能命中相当一批种子用户群。
      好的生产工具也是会促进用户的生产行为的。
      比如买车了，就更愿意折腾和出门；有了好的编辑器，也会激发码字和记录的欲望。
    1. 用户对收费极度敏感。
      Evernote等软件产品路线并不完备，没有额外的用户规模增长点，用户会员和企业会员是主要的收入来源。因此受限于盈利压力，阉割功能，来让用户掏钱。
    1. 校园课程笔记
    1. 校园环境下，学生获取课件的方式依然是U盘，网盘，或者QQ群，班级群邮箱等方式。

    调研的学生群体里，清华在使用快手联合开发的雨课堂，其敷衍程度，一言难尽。北航的学生的获取方式疫情期间有比较大进步，这是一个非常官方的说法。

    1. 课件内容也是万年不变的ppt，pdf和word。这样的文件类型是没有fork价值的。

    老师们还在用着五到十年制作前的课件，所以需要重新以新标准格式化。
    老师是万万不可能有动力去帮学生重新格式化新的课件的。
    可以花一笔预算，来雇佣学生以Markdown的格式编辑相应的课件和复习资料。

    这个成本并不高。而且很多上过网络公开课的人都有这种分享习惯。在Bilibili或者YouTube学习MIT, Stanford等学校的计算机课程和数学课程，并记录课程笔记，而且在github上分享相关的课程笔记，格式也是Markdown。其star数都是几千以上。

    1. 既然有普通用户自发的分享课程笔记的行为，且star数普遍上千。说明这种用户需求是存在且强烈的。那自然可以用商业化的组织行为，批量的制作课件，免费的分享给目标用户群，来获取冷启动用户。
    2. 中国大学生群体数量庞大。中国有近600万程序员，全球程序员不超过1亿。中国仅大学生群体人数就数千万之多。
    1. 泛化与扩张
    1. NoteFeed 本身形式的扩张。

    合理的产品设计模式，不会产生突兀感。笔记应该是可以泛化为纯文字、富文本(支持latex)、图文、视频等多种Feed形式。

    1. 内容体系的扩展
    1. 校园用户的课件和复习笔记只是单点的切入口，如何由此扩展到学生群体的其他衣、食、住、行、娱乐、消费等相关领域，非常重要。

    这里的衣食住行可以理解为关于衣食住行的笔记攻略。
    比如美妆教程，美食攻略，旅游功率，消费攻略等。
    也可以理解为用户的个人笔记或者教程。
    比如基于图文的如何制作美食的笔记。

    1. 用户人群的扩张
    1. 程序员群体也有非常强烈的笔记诉求，如 掘金小册。
      程序员经常百度或者Google到重复的错误技术博客。就是因为缺少fork的机制，所以常常出现错误的技术文章被反复复制和转载。有记笔记习惯的程序员会把一些重要的东西花时间整理到自己的blog里去，但是更多的时候只是单纯的把一个技术博客添加到自己的收藏家书签里。
    2. 各大垂直社区的用户群和博主
      比如小红书，马蜂窝，下厨房，甚至知乎，微信公共号。当流量起来之后，也能吸引到部分人群入驻。
      如果凭借底层Feed设计的高度统一以及持续强工具属性，成为强粘性的平台后，会源源不断虹吸各家的流量入驻。
    1. 用户行为模式的扩展

    随着用户接受度的提高，用户养成在平台里码字和制作图文、图集、视频与朋友分享或者记录生活的习惯。

  4. 成功案例

    1. github

    github.com 是全球最大的同性交友社区。

    1. git 是一个分布式版本控制系统。这种系统虽然可以处理任何文件内容。但是天然对纯文本的内容最为友好。无论是笔记还是代码，都是以文本内容为最重要的载体而存在。
    2. git fork 的机制，在fork 图片和 fork 视频的时候是没有意义的。图片和视频的排他性很强，越是有人出现的图片和视频，排他性心理越强。
    1. Pinterest

    Pinterest 是使用Pin的机制搜集网络上的图片到个人的图片库中。和 git fork 原理不同。
    共同点是都获得一份复制对象。
    Pinterest上用户更多的是Pin卡通、风景、剪纸、插画等艺术风景类图片，也符合前面讨论的fork图片排他性心理，即人们在fork其他人的自拍类的图片时候，抵触感很强。

    因为是基于人为中心设计的社交网络，用户在Pin或者fork的时候，会非常在意内容的认同归属问题。

风险

  1. 用户笔记ugc或分享意愿不强烈

    产品设计需要非常警惕。尤其是要处理好隐私和公开笔记的分离问题。
    产品本身的工具易用性属性也会严重影响用户的生产行为，可正可负。
    笔记内容的泛化程度也严重影响用户的参与行为，可正可负。

  2. 流量池太小

    笔记的总流量池大小无法与短视频等平台的流量池大小相提并论。
    即所有品类的笔记流量总和远小于抖音快手对应的视频娱乐流量。

  3. 来自竞争对手的反击与遏制

    这种知识组织和分享的方式，如果做起来了，会对知乎和公众号等平台产生强烈的冲击，分流用户使用时长。
    内容的生产者和用户的社交行为是跟着流量平台走的。
    谁的流量大，谁的ugc成本就更低，质量更高，社区也更活跃。
    趟过总流量门槛的阈值，很重要。

花絮周边与QA

@(为什么是 Markdown？)

不同的时代都会有相适应的新标准出现。Word是上世纪80年代的产品，那个年代浏览器和HTTP协议(89年HTTP 0.9诞生，93年HTTP 1.0稳定)都没有出现。基于IT产业向前兼容的设计准则，word和pdf等格式是会被浏览器甚至移动端兼容的。flash没有那么幸运，被巨头一起做掉了。野生的JSON和Markdown的流行都是开发者用脚投票的结果。Markdown已经成为纯文本格式编写文档的事实标准并有强大的周边生态。换言之，如果Word和Pdf是诞生在2000年之后的文件格式，或许也会使用类markdown的标准来定义图文的排版、布局、显示与扩展。

当Markdown不能满足越来越多的场景需求时，也需要考虑扩展Markdown格式。技术标准应该为产品场景服务。

@(为什么要强迫用户用Markdown编写？)

不能无底线的惯着用户，用户需要学习来提升整体的体验。
不用Markdown并不妨碍用户使用笔记，但是如果希望得到精致美观多样化的呈现效果，用户需要学习。
Markdown的学习成本会造成一部分用的流失，可以考虑通过其他方式来降低Markdown的学习和使用门槛。

@(课程格子 和 超级课程表 为什么没了？)

课程格子和超级课程表曾经都依靠 课表 这个单点功能打入校园市场，并获得百万级种子用户。但课表本身是一个非常单薄的工具，并不能支撑长久的价值，也没有足够的壁垒留住用户。这种工具社交化的压力既来自于投资人商业逻辑，也来自于产品团队本身渴望进一步覆盖更多用户场景。

课程格子坚持了产品原则，即课表功能的产品不应该加入花里胡哨的社交化功能，这是符合用户预期的。然后后续投资谈不下来，资金链断裂，产品被放弃。
超级课程表，希望走出校园社交的路。但是一个从一开始就缺乏统一规划，全部的功能位都是围绕一个简单课表来铺开的体系，是无论如何都容不下社交的场景的。最后强行内嵌一个BBS进去，产品形态撕裂。用户用脚投票放弃了产品。

@(gihub 发展历程)

github 是三个程序员兼职写出来的，因为还需要打零工来养活团队。最初的版本，没投资，没用户，没啥用。由于github创始人和Ruby on Rails的创始人认识，Ruby on Rails的包管理也需要一个包托管平台，选中了github来作为代码和包托管平台。后来nodejs横空出世，npm作为node官方推荐包管理器，再一次选中了github作为代码托管平台。node的用户群规模是非常庞大的，远远大于ruby社区的人。github迎来了高速增长的时代。

自此，iOS，PHP，Python，C/C++, Java等各个语言系的开发者都从四面八方入驻到github。github前时代的 google code 凉， sourceforge 凉。

由RoR和node掀起的包管理器云时代，其背后的“云”的开发管理逻辑，是github的最大幸运。

@(为什么小红书转型短视频了)

小红书早期是做购物笔记起家，但是这个垂直市场本身很小，天花板来的很快，故而小红书一直在不断寻求转型和突破。从导购攻略→\to→种草图文社区→\to→海淘代购→\to→女性短视频社区→\to→年轻人的生活平台(？？？)。第三、四个阶段间的日活数据在200w左右，全面转短视频之后才获得新生。

小红书放弃了全品类的笔记获得新生，不能完全否定笔记产业本身的规模和价值。只是产品路线选择不同。

@(战略 与 战术，工具 与 社区 )

战略的实现通常都依赖配套的战术体系。古代由于知识的垄断，有意识谈战略的人都是手握战术资源的人。现代社会由于信息传播越来越短、平、快，战略已经是一个人人都能高谈阔论的东西了。但是只有手握执行战术所需资源的人才有资格去实践战略。

同理，工具 与 社区 也是相辅相成的关系。买量预算特别足或者自家流量特别多的团队，可以跳过工具积累的阶段，直接正面硬刚社区壁垒。用户，KOL，内容，全部都可以用钱和流量来搞迅速搞定。

钱和流量是搞定这个难题的必要条件，不是充分条件。

买量预算不足或者产品原创能力特别强的团队，可以选择另一种平滑的打法。基于产品特征设计工具属性，即提供的工具属性要和产品将来的社区化进化路径保持一致，以此切入目标用户市场，获取种子用户，并逐步社区化。

产品创新的本质也是为了获取流量。产品创新获取社交网络流量的成本要远远低于花钱买量，而且口碑和品牌性都更好。

产品创新本身的成本确是非常非常非常非常高。
直接花钱买量可以是一个简单持续的过程，但是创新不是。可以称得上创新的东西，积累、筹备的周期很长。三年不鸣，一鸣惊人；三年不飞，一飞冲天。
